很长一段时间以来，**跨设备文件传输**始终是一个看似简单却时常刺挠我的痛点。为了将一张图片传到电脑中的指定文件夹中，我先从相册中分享图片到微信传输助手，然后打开微信的电脑端，将这张已经被下载好的图片又重新另存到我想要的文件夹中。但其中每个环节都需要较长的步骤实现，还担心点错，还存在压缩。

为了追求极致的纯粹与高效，我构建了一个基于python 的极简传输方案。它摒弃了冗余的 UI 和复杂的配置，回归到了 **Web 交互最本质的 Client-Server 架构**，也帮助我更深刻的理解了计算机网络通信的行为。

## 核心交互流程解析

这一工具的运作逻辑可以高度压缩为以下五个阶段：

1. 服务的初始化与“发现”。服务器（电脑端）启动后，首先通过诱导网卡连接公网 DNS 的方式，精准**捕获本机的局域网物理 IP 地址**。随后，程序利用 qrcode 库将该 IP 与监听端口（如 5000）封装成一个二维矩阵。
2. 握手与请求。客户端（手机）扫码后，浏览器发起 TCP 三次握手，随后发送一个 HTTP GET / 请求。此时，电脑端的 Flask 路由被触发，它像一个守门人，精准地匹配到预设的根目录逻辑。
3. 资源的物理迁移。服务端执行 `render_template`，将硬编码在 Python 脚本中的极简 HTML 模板注入变量。**这一段前端代码并非在服务器执行，而是转化为字节流（Byte Stream），跨越路由器和网线，最终抵达手机浏览器的内存空间**。
4. 客户端渲染与环境隔离。**手机浏览器接收到报文后，其渲染引擎开始解析 DOM 树与 CSS 样式**。此时，一个临时的、具备交互能力的“控制面板”已经在手机端构建完成。得益于 JavaScript 的异步特性，手机端已准备好发起后续的 POST 文件传输请求。
5. 闭环：双向数据流。当用户在手机上点击“上传”时，二进制文件被切片并封装进 `FormData `对象。服务器端的 upload 路由接收并将其持久化存储到指定的 uploads 文件夹中。至此，一个从“发现”到“交付”的完整信息熵减过程宣告完成。

## 与公网环境下的对比

1. 寻址：我的局域网通过扫描二维码获取内网ip，而公网通过**DNS域名解析**获取公网ip。
2. 连接：局域网是HTTP明文直连，而公网是**HTTPS，利用TLS加密隧道。**
3. 路由：局域网只需在路由器内单跳完成，公网需要经过多个骨干网路由器及CDN节点。
4. 服务端：我的局域网服务端是单机单进程Flask，而公网的服务端利用了**负载均衡器和集群服务器**。
5. 存储：我的局域网存储在本地uploads文件夹，而公网的是**云端分布式对象存储**OOS。

## 结语

这个工具的想法起源于上学期学习计算机网络过程中的一个课程作业：实现一个两个节点的通信。当时我实现了一个基于SQL Server 的前后端分离的聊天系统，还借助 #VPN （虚拟专用网络）实现了和一位武汉朋友的通信。但当时做的无论是界面还是代码结构都实在粗糙，最关键的是，我需要先把前端文件发给他才行。但现在回看，确实走了不少弯路。由此可以看出对于知识的理解需要有一段沉淀期的，如果你目前也在对正在学的知识感到困惑，不妨先放一放，时间会让你更理解。
