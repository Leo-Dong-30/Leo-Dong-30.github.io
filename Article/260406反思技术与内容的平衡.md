昨天一直在改进博客的代码和自动化的脚本，关于使用 #AI编程 有了更多的经验，也产生了关于 #技术 和内容的辩证思考。

## 项目

在vibe coding之前，**先要有一个项目的概念**。一个项目简单来说就是一个文件夹，所有代码、文档、资料都存放在这里面。不同类型的代码最好放置在同一个项目下不同的文件夹中，代码间通过相互的引用传递函数等。我之前vibe coding的时候，生成的代码总是存放在同一个文件里面，这样导致文档太长，结构混乱。最麻烦且常见的情况是，你只想要改动一小块功能，但因为上下文过长，AI 总是会把不该改动的地方也改了（即幻觉），就导致改了一个问题，又出来很多个问题的情况发生。所以最好能在项目开始前确定好大致的项目框架（项目树）。每当完成了较大改动后，及时对代码进行重构。

## 项目环境管理

第二个是**项目环境管理**的问题。以python为例，我总是会遇到明明下过同样一个库，但在运行别的项目的时候又要下载一遍的问题，现在我知道了是环境路径混乱的问题。我总结了几点经验：

- 永远不要把具体项目的第三方库装在系统的全局环境（如 C 盘的 Python 或 Anaconda 的 base）中。全局环境是所有项目都可以用的，但是不同项目需要的版本不同，因此很容易造成开发环境混乱。要养成“新项目 = 新 venv”的习惯，在命令行输入 `python -m venv .venv`，在vscode中选择 `.venv\Scripts\python.exe`
- 很多时候在 VS Code 里选好了解释器，但右下角的终端面板还是旧的（显示 (base) 或者什么都不显示），因此实操中要彻底关闭旧终端，开启新终端。只有看到命令行开头出现了 (.venv)，你输入的 `pip install` 才是安全的。

## 文章向量化

第三个是关于这个博客本身的。我使用了一个 #transformer 模型（MiniLM）来对我的文章进行处理。这个模型的工作本质是：**先通过深度注意力机制提取文本特征，再将复杂的自然语言压缩成一个固定维度的数值向量，最后通过计算余弦相似度来量化文章之间的“亲疏关系”**。由于模型有token输入限制，因此需要将长文章切片，**分别计算向量再取平均值**，这保证了长博文的语义不会因为被截断而丢失。关于这一点，我其实是不太满意的，一直想换一个功能更强大的模型，可以直接读取全部的文章。从昨天下午开始一直在折腾，报错，改错如此循环，到最终也没能用上所谓更优秀的模型。此外有了向量化的文章之后我还可以做很多事情，比如在阅读文章底部做相关文章推荐 #推荐系统 ，比如绘制全局的相关热力图而不仅限于一定阈值之上的连线，还可以做一个时间轴，滑动时间块展示文章的生长过程……

这些奇思妙想也促使我反思博客技术与内容的平衡，因为系统2（思考，快与慢）告诉我，这些现在都太悬浮了，没有足够的文章作为积累，一切皆为浮云。一方面过度追求最新最好的技术很有可能让我陷入工具思维，反而忽视了真正重要的资产其实是这一篇篇的文章。这和我之前折腾笔记软件很类似，我尝试过很多软件，每个软件都有自己的优势和不足，但我的欲望是无限的，抱着准备好了再上路的心态确实不是能成事的人该有的，这点我需要反思。另一方面，这也倒逼我写博客的时候不要口水话，不要过度发散。因为MiniLM模型只有300多的维度，而且还是分段平均处理的，如果一篇博客没有一个明确的主旨和简洁凝练的语言，模型就会自动将这篇文章判为“**平均值文章**”，毫无特色，泯然众人。这的确是一个很好的监督机制。
